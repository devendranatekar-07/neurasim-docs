Tablet TCP Communication

Complete reference for the bidirectional TCP protocol between VR headset and therapist tablet — discovery, connection lifecycle, wire framing, message routing, and threading model.

1. Overview

TabletCommunication is a persistent singleton that manages all real-time communication between the VR headset and the therapist's companion tablet. It runs two TCP servers and uses UDP for network discovery.

Active only in Stationary (Tablet) Mode
When the device is configured for home therapy, both TCP servers are stopped and the app communicates directly with the backend REST API. The TCP layer is exclusively for the stationary clinic workflow where a therapist drives therapy from a tablet.
Primary Responsibility

Bidirectional session between headset and tablet. Tablet sends therapy commands, game configs, and credentials. Headset sends telemetry, game results, and session completion.

Architecture Pattern

Two separate TCP server instances on different ports (one queued outgoing, one real-time incoming) plus async UDP discovery broadcasting every 3–5 seconds.

Persistence

Survives all scene transitions. DestroyedOnSceneChange() returns false. Created at app startup, destroyed only when the application exits.

Threading

All TCP I/O runs on background threads. Callbacks are marshalled to the Unity main thread via UnityThread.executeInUpdate() before touching Unity APIs.

Why Two Separate TCP Servers?
The split design enforces unidirectional ownership. The listener (8503) accepts one tablet connection and never sends from it. The sender (8502) has an outgoing queue to decouple game completion events from network timing. This prevents game completion callbacks from blocking if the tablet is momentarily slow.

Async Shutdown Requirement
StopTabletTcpServ() is async UniTask and must be awaited. In ConnectServerWindow it is called as await TabletCommunication.Instance.StopTabletTcpServ(). Calling it with .Forget() is done only from OnDestroy() where no await is possible. If shutdown is not awaited, the next StartServer() may race against incomplete cleanup.

5. Wire Protocol

5.1 Frame Format

All TCP messages — in both directions — use the same simple length-prefix framing. There are no delimiters or packet boundaries beyond the size header.

// ─── Frame Layout ───────────────────────────────────────────
//
//  ┌────────────┬─────────────────────────────────────────────┐
//  │  2 bytes   │           N bytes                           │
//  │  UInt16    │           ASCII JSON                        │
//  │  big-endian│           payload                           │
//  └────────────┴─────────────────────────────────────────────┘
//
//  Size is the byte length of the ASCII payload (not chars).
//  ARM big-endian: byte order is REVERSED before sending.
// ────────────────────────────────────────────────────────────

async UniTaskVoid SendStringFrame(string data)
{
    byte[] payloadBytes = Encoding.ASCII.GetBytes(data);
    UInt16 msgByteSize  = (UInt16) payloadBytes.Length;

    byte[] sizeBytes = BitConverter.GetBytes(msgByteSize);
    Array.Reverse(sizeBytes);   // big-endian for ARM

    byte[] frame = sizeBytes.Concat(payloadBytes).ToArray();
    await networkStream.WriteAsync(frame, 0, frame.Length);
}
Why ASCII and not UTF-8?
All JSON payloads are restricted to ASCII-safe characters. Therapy parameters, game IDs, session tokens, and device identifiers are all numeric or Latin alphanumeric. Using ASCII avoids multi-byte encoding surprises in the size calculation and keeps the framing simple.

8.2 UnityThread

Assets/Shared/Scripts/UnityThread.cs

A lightweight main-thread dispatcher that bridges background TCP threads to the Unity main thread.

// Thread-safe enqueue from any thread:
public static void executeInUpdate(Action action)
{
    lock (actionQueuesUpdateFunc) {
        actionQueuesUpdateFunc.Add(action);
        noActionQueueToExecuteUpdateFunc = false;
    }
}

// Main thread drain (called every frame in Update()):
void Update()
{
    if (noActionQueueToExecuteUpdateFunc) return;  // fast path
    lock (actionQueuesUpdateFunc) {
        actionCopiedQueue.AddRange(actionQueuesUpdateFunc);
        actionQueuesUpdateFunc.Clear();
        noActionQueueToExecuteUpdateFunc = true;
    }
    foreach (var action in actionCopiedQueue) action.Invoke();
    actionCopiedQueue.Clear();
}

The volatile noActionQueueToExecuteUpdateFunc boolean acts as a fast-exit guard so the lock is only acquired when there are actions waiting — reducing contention on frames with no TCP activity.

9.2 OTA Update Pause / Resume

During an OTA software update, the TCP servers are stopped to avoid interference. They are automatically restarted after the update completes — but only if the app is currently in Tablet mode.

// Subscribed in Awake() — persists for TabletCommunication lifetime
_subscription = Broker<UpdatingStateChanged>.Subscribe(
    (object sender, in UpdatingStateChanged args) =>
    {
        if (args.IsUpdating)
            StopTabletTcpServ().Forget();   // pause during install
        else if (GlobalConfig.communicationMode == CommunicationMode.Tablet)
            RunTabletTcpServ();              // resume after install
    });

11. Known Issues

Hardcoded Telemetry (All Battery / Controller Fields)
SendTelemetry() sends hardcoded values (70% headset battery, 80% controllers, true for pair_controller_0) to the tablet. The real data reads are commented out. This means the therapist's tablet always shows the same dummy values in the headset status panel. The BatteryLevelWatchdog monitors real battery levels internally, but this data is not reflected in the telemetry sent to the tablet.
WiFi Hotspot Password in Plaintext
ConnectTabletWifi.cs contains a hardcoded WiFi password:
private const string wifiPass = "R3mm3D#!";
This is used to connect to a tablet hotspot. The password is committed in source code and visible to anyone with repository access. Consider moving to a configurable credential or StreamingAssets config file.
Thread.Sleep(2000) on TCP Receive Thread
When a therapy.enter == true command arrives, the code calls Thread.Sleep(2000) on the TCP receive background thread before calling AndroidHelper.WakeUpHeadset(). This blocks the listener thread for 2 seconds. During this time, no other incoming messages on port 8503 can be processed. This is a workaround to ensure the headset is fully awake before handling the Enter command.
TherapyEvent() Marked [Obsolete]
LoaderController.TherapyEvent() is decorated with [Obsolete("Use TherapyEventBroker.InformSubscribers")] but is still the actual entry point called from the TCP receive path. The "legacy" fallback OldTherapyEvent() is still invoked for compatibility with older code paths.
