Login Flow

Complete authentication flow from app launch to therapy-ready state — device auth, WiFi, TOTP, TCP servers, error handling, and all failure modes.

1. Overview

NeuraSim VR (product name: BeeVee) uses a device-centric authentication model. There are no user credentials (username/password). Instead, the VR headset itself is the identity — its serial number is pre-registered in the NeuraSim backend by clinic administrators.

Authentication Model Summary
Auth TypeDevice-based (headset serial number)
CredentialDevice UUID (Pico VR serial) + TOTP token + static API key
EndpointPOST /web/login
Session TokenJWT-like token_session.token returned on success
Therapy Modesstationary (tablet-controlled) or home (standalone)
No User Input RequiredLogin is automatic once WiFi is established

3. Phase 1: App Boot & Initialization

The application starts in the Loader scene (Assets/InitialScreen/Loader.unity). Two key MonoBehaviours kick off the initialization sequence:

3.1 LoaderController Initialization

Assets/LOADER/Scripts/LoaderController.cs

LoaderController is a persistent singleton (SingletonMonoBehaviour<LoaderController>) that survives scene transitions. It is the central orchestrator for the entire application lifecycle.

1
LoaderController.Awake()

Called once when the Loader scene loads. Performs critical initialization in this order:

// Simplified Awake() sequence

3.2 GlobalConfig Initialization

Assets/Shared/Scripts/Shared Scripts/GlobalConfig.cs

GlobalConfig.Init() is a critical static initializer that sets up the entire server configuration. Here is what happens:

A
Load or create server config file

Reads serverConfig.json from Application.persistentDataPath. If this is the first run after installation (detected by comparing Application.buildGUID against PlayerPrefs["BUILD_GUID_VERSION"]), copies the config from StreamingAssets instead.

B
Construct server URL

Currently hardcoded to https://remmedvr-dashboard.neurasim.health/ regardless of region/instance settings (the dynamic URL construction exists but is overridden at lines 304-308).

C
Configure NetworkStatusWatchdog

Sets up two health-check endpoints: https://google.com (HEAD, 200) for internet reachability, and the server URL (HEAD, 200) for server reachability. These are polled every 10 seconds.

D
Restore persisted settings

Loads communicationMode, userLoggedFirstTime, installationTargetDevice, and HandNess from PlayerPrefs.

Server URL Note
In CreateUriFromConfig(), the region/instance switch-case logic is present but overridden at the end of the method. All builds currently point to https://remmedvr-dashboard.neurasim.health/. This is likely a temporary hardcode.

3.3 WebApiPass Initialization

Assets/Shared/Scripts/WebAPI/JsonArraySerializer.cs (contains WebApiPass class)

WebApiPass is the credential store used by all subsequent API calls. On init, it creates a singleton instance with the hardcoded API key:

public WebApiPass()
{
    api = new Api();

After login, the device, session, and token_session fields are populated from the login response and used for all authenticated API calls.

4. Phase 2: WiFi Connection

Assets/InitialScreen/Scripts/PreInitController.cs

PreInitController.Start() calls InitWaitingRoom(), which orchestrates WiFi connectivity before login can proceed.

4.1 Auto-Connect Attempt

private async UniTaskVoid InitWaitingRoom(int wifiAttempts = 20)
{
    LoadingCircle.Instance.Show();
    // Show "Waiting for WiFi..." toast

    if (!AndroidHelper.IsWifiConnected())
    {
        for (int i = 0; i < wifiAttempts; i++)  // 20 iterations, 1s each
        {
            var configuredWifis = AndroidWifiManager.GetConfiguredNetworks();
            var currentConfig = configuredWifis.FirstOrDefault(
                w => w.status == Status.ENABLED);
            if (currentConfig != null)
                AndroidWifiManager.Connect(currentConfig.SSID, currentConfig.preSharedKey);
            await UniTask.Delay(1000);
            if (AndroidHelper.IsWifiConnected()) break;
        }
    }
}

5.2 TOTP Token Generation (rkey)

Assets/Shared/Scripts/WebAPI/Request.cs:14-29

The rkey is a Time-based One-Time Password with a 60-second window:

public static string GetRKey(Int64 secret)
{
    Int64 epochSeconds = (Int64)(DateTime.UtcNow - new DateTime(1970,1,1)).TotalSeconds;
    Int64 c = epochSeconds / 60;            // Time window (60 seconds)
    Int64 j = c * secret;                    // secret = 192512341
    Int64 p = j % 1000000;                  // Modulo to 6 digits
    string hash = SHA256Hash.ComputeSha256Hash(p.ToString());
    return hash.Substring(0, 6);           // First 6 chars of SHA-256
}
Clock Sync Requirement
The device's system clock must be within ~60 seconds of the server's clock. If the clocks are out of sync, the TOTP token will be invalid and the server returns "invalid_rkey", which maps to error code 11 (Time_Out_Of_Sync).

5.4 Login Response

On HTTP 200, the response is deserialized to WebRequest.ShortLogin.ResponseBody:

public class ResponseBody
{
    public string modeTherapy;        // "home" | "stationary"
    public Assignment assignment;      // Patient-device assignment
    public Device device;              // Device record (idkey, pin)
    public Clinic clinic;              // Assigned clinic info
    public User user;                  // User/patient data
    public int ln;                     // Login count (1 = first login ever)
    public DevicePairing device_pairing; // device.idkey + session.sessionid
    public TokenSession token_session;  // JWT-like token for subsequent calls
    public string error;               // null on success, error msg on failure
    public int num_error;              // 0 on success, LoginErrorCodes on failure
}

Success Determination

if (loginResponse.error is null)
{
    // SUCCESS PATH
    // 1. Validate modeTherapy is present and valid
    // 2. Validate clinic is assigned
    // 3. Parse therapy mode (home/stationary)
    // 4. Check first-time login (ln <= 1)
    // 5. Branch by therapy mode
}
else
{
    // ERROR PATH → Show ServerErrorMessageBox
}

6. Phase 4: Post-Login Branching

After successful login, the application branches based on loginResponse.modeTherapy:

6.1 Stationary Mode (Therapist-Controlled)

STATIONARY Clinic-based, tablet-directed therapy
1
Set communication mode

GlobalConfig.SaveCommunicationMode(CommunicationMode.Tablet) — saves to PlayerPrefs

2
Display clinic name

Shows loginResponse.clinic.name on the Stationary UI panel

3
Start tablet TCP servers

TabletCommunication.Instance.RunTabletTcpServ() — starts listening on ports 8502 (sender) and 8503 (listener). Begins UDP broadcast discovery on port 8505 every 3s.

4
Complete

After 2s delay, shows "Connected successfully" and calls CompleteModule() to advance the SlideMenu.

Tablet Credential Handoff
In Stationary mode, the device, session, and token_session credentials are NOT stored from the login response directly. Instead, they arrive later via the tablet's init_data TCP message through LoaderController.InitWebAPIData().

6.2 Home Mode (Patient Self-Directed)

HOME Patient home use, standalone operation
1
Set communication mode

GlobalConfig.SaveCommunicationMode(CommunicationMode.Standalone)

2
Display patient info

Shows username, patient name, specialist name/surname, and clinic name from the login response.

3
Stop tablet servers

TabletCommunication.Instance.StopTabletTcpServ() — no tablet needed in home mode.

4
Store credentials

Builds InitData from loginResponse.device_pairing and loginResponse.token_session, then calls LoaderController.Instance.InitWebAPIData(initData, Standalone) to populate WebApiPass.

5
Resend any unsent therapy data

HomeTherapySession.CheckAndResendPreviousTherapy() — ensures previously completed but unsubmitted sessions are sent.

6
Fetch therapy details

POST /therapy/get with device, session, api, and token_session credentials.

7
Display therapy info & check limits

Shows completed sessions vs daily limit, therapy date range. Checks if daily session limit is reached and if the therapy stage has completed (last day + all sessions done).

8
Complete

Shows "Therapy Data Acquired", then calls CompleteModule().

6.3 First-Time Login Detection

// First-time login is determined by login count from server
if (loginResponse.ln <= 1 && therapyMode == TherapyMode.home)
{
    loggedFirstTime = true;
    GlobalConfig.SaveFirstLoginFlag(isFirstLog: true);
    HomeTherapySession.ClearUnsentTherapyData();  // Clean slate
    DemoTracker.Instance.ClearPlayedDemos();     // Reset demo history
}

When userLoggedFirstTime is true, the SlideMenu continues to Tutorial/Handedness screens instead of closing immediately after login.

What is DemoTracker?
DemoTracker is a persistent singleton that records which game demos the patient has already watched. On first login it is cleared so the patient sees intro demos for each game fresh. On subsequent logins it persists so they are not shown the same demos repeatedly.

7. Phase 5: Ready State

After ConnectServerWindow.CompleteModule() is called:

// In SlideMenu.ContainerCompletedCallback():
if (currentMenuContainer is ConnectServerWindow)
{
    if (!GlobalConfig.userLoggedFirstTime)
    {
        Close();  // Returning user → close wizard immediately
        return;
    }
    // First-time → continue to next wizard screens (Tutorial, etc.)
}

After the SlideMenu closes, PreInitController.SlideMenuCompleteModule() is called, which invokes SwitchUIBasedOnTherapyMode():

Therapy ModeUI ModeWhat Happens Next
stationary LoaderMode.Stationary Shows stationary UI. Waits for therapist to send "Enter" command from tablet.
home LoaderMode.Home Shows home UI with "Enter VT Office" button. Patient taps to enter Hub.

The "Enter VT Office" button (PreInitController.EnterVTOffice()) performs final checks before entering the Hub:

  • NetworkStatusWatchdog.ConnectionState == TargetServerReachable — server must be reachable
  • BatteryLevelWatchdog.Instance.CheckBatteryLevels() — headset and controller batteries above minimum threshold

If either check fails, a blocking MessageDisplay dialog appears with an appropriate error. The patient cannot enter the Hub until both conditions are satisfied. BatteryLevelWatchdog is a persistent singleton monitoring battery levels continuously throughout the session.

8.2 Network-Level Error Handling

Assets/Shared/Scripts/WebAPI/RestAPI.cs

The RestApiCall.PostRequest<T> method handles HTTP-level errors:

500Save server error HTML to persistentDataPath/serverError.html. Invoke errCallback with code 500.

For the login call, retryOnError: false — the request does NOT auto-retry. Instead, errors are shown in the ServerErrorMessageBox which has a retry button.

10. Dependencies

10.1 Runtime Dependencies

WiFi Connection

Required before login. Checked via AndroidHelper.IsWifiConnected(). Auto-connect uses FSG.Android.Wifi.AndroidWifiManager.

NeuraSim Server

Must be reachable at https://remmedvr-dashboard.neurasim.health/. Monitored by NetworkStatusWatchdog (HEAD request every 10s).

Pico VR SDK

Pvr_UnitySDKAPI.System.UPvr_GetDeviceSN() provides device serial number. PicoMobileSDK must be initialized.

System Clock

Device time must be within ~60s of server time for TOTP token validation. Uses DateTime.UtcNow.

BatteryLevelWatchdog

Persistent singleton monitoring headset and controller battery levels. Blocks entry to the Hub (EnterVTOffice) if batteries are too low. Fires Broker<BatteryLevelChanged> events.

LitJson

JSON serialization/deserialization for all API communication. JsonMapper.ToJson() / JsonMapper.ToObject<T>().

UniTask (Cysharp)

Async/await framework. Used in WiFi polling, post-login delays, and non-coroutine async flows.

10.2 Singleton Initialization Order

These singletons must be alive before login proceeds:

  1. LoaderController — initializes GlobalConfig, WebApiPass
  2. NetworkStatusWatchdog — configured by GlobalConfig.Init()
  3. TabletCommunication — started/stopped based on therapy mode
  4. LoadingCircle — UI feedback during login
  5. ToastText3D — toast notifications
  6. LocalizationManager — provides localized strings

14. Security Notes

Observations for Security Review
  • Static API Key: The API key s4dfghjiko0j9h8g7f6d is hardcoded in the client binary. It is the same for all headsets and provides no real security by itself.
  • TOTP Shared Secret: The secret 192512341 is also hardcoded. The 60-second TOTP window mitigates replay attacks but the secret is extractable from the APK.
  • HTTPS: All API communication uses HTTPS. The ForceAcceptAll certificate handler exists but AcceptAnyCertificate defaults to false (strict SSL enabled by default).
  • Device-Only Auth: Physical access to a registered headset grants full therapy access with no additional authentication. This is by design for patient usability.
  • Token Persistence: The token_session.token is held in memory only (WebApiPass) and not persisted to disk, so it is lost on app restart.
