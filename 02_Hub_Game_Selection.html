Hub & Game Selection

3D lobby, portal interactables, demo enforcement, and therapy game launch flow

Overview

The Hub is the 3D lobby scene that sits between game sessions during an active therapy. It is loaded by LoaderController after therapy entry (TherapyEventFlags.Enter) and again after each game completes, until the full session is done.

What the patient sees and can interact with in the Hub depends entirely on the therapy mode. In stationary mode the Hub is purely decorative — game selection is driven by the therapist's tablet. In home mode the Hub is interactive — physical game portals appear and the patient must watch a demo of each game before starting therapy.

Entry Gates

Before transitioning to the Hub, two watchdogs are checked:

NetworkStatusWatchdog

Verifies active internet + reachability to the server URL. If the network check fails, an error dialog is shown before Hub loads in stationary mode.

BatteryLevelWatchdog

Monitors headset battery. A low-battery warning may be shown. Does not block Hub entry but fires Broker<BatteryLevelChanged> events that UI components listen to.

Hub Modes

The Hub behaves differently depending on GlobalConfig.communicationMode and LoaderController.currentTherapyMode:

stationary

Stationary Mode

Full clinical setup — therapist controls everything from the companion tablet.

  • No portal prefabs are spawned (InteractableManager.Start() returns early)
  • SettingsDoorInteractable is permanently blocked — patient cannot exit
  • StartTherapyButton is hidden (SetActive(false))
  • Hub simply waits for a TherapyEventFlags.StartGame event from the TCP listener
  • Parameter polling (TherapySetParameters) begins immediately on game start
home

Home Mode

Patient uses the headset independently at home — no tablet required.

  • InteractableManager spawns one portal DemoPlayButton per game in the therapy plan
  • Patient must watch a demo (max 3 plays per game) before therapy unlocks
  • StartTherapyButton is visible but disabled until all mandatory demos are complete
  • 15-second timer runs between games (HomeTherapyDelayedNextGame)
  • SettingsDoorInteractable is enabled — patient can exit to Loader/settings
Hub entry parameter: StartHub(bool enableInteractables) — when called with true, immediately fires Broker<InteractablesStateChange>(0f) to unblock portals. Called with false during SessionComplete to keep Hub locked while data uploads.

AsyncLoadScene()

public async UniTask AsyncLoadScene(string name, LoadedSceneCallback loaded = null,
    SceneParameters? sceneParameters = null)
{
    LoaderController.sceneParameters = sceneParameters ?? SceneParameters.Default;
    AsyncOperation asyncLoad = SceneManager.LoadSceneAsync(name);
    if (asyncLoad != null)
    {
        progressBar.gameObject?.SetActive(true);
        CancelInvoke();                              // cancel any pending Invoke callbacks
        UGC.UGCGameController.Instance?.OnGameStop(); // stop any running game
        while (!asyncLoad.isDone)
        {
            progressBar.Value = asyncLoad.progress * 100f;
            await UniTask.WaitForEndOfFrame();
        }
        await asyncLoad.ToUniTask();
        loaded?.Invoke();                            // callback fires AFTER scene is live
    }
    else
        ToastText3D.Instance.Show3DTextToast("Scene is missing");
}

The progress bar is driven from AsyncOperation.progress (0→1 mapped to 0→100%). UGCGameController.OnGameStop() ensures the previous game is cleanly stopped before the new scene loads. The loaded callback fires after the scene is fully active.

Interactables (Home Mode)

All interactive Hub objects implement InteractableObject (base class) and subscribe to Broker<InteractablesStateChange> for block/unblock events. The state change carries a duration in seconds; zero or negative unblocks immediately; InteractablesStateChange.Permanently blocks indefinitely.

InteractableManager

Assets/LOADER/NewLoaderGraphics/interactable objects/InteractableManager.cs

Runs at Hub scene startup. Dynamically spawns the correct game portal prefab for each game in the current therapy session set, placed at numbered spawn points.

1
Checks LoaderController.currentTherapyMode == TherapyMode.home. In stationary mode, logs "dont spawn frames" and exits early.
2
Iterates currentTherapy.sessionset.Last().sessionsetgame. For each game, picks the matching frameObjects[n] prefab by game_id and instantiates it at the next framePositions[i] spawn point (sorted by name). Sets DemoPlayButton.instanceID to the session-set game's unique id (not the game_id).
3
Queries DemoTracker for any games with zero plays. If any exist, starts the ForceDemo() coroutine which blocks all interactables, unblocks the settings door, and individually enables colliders only on the unplayed portals (with forcedInteraction=true). Games already played at least once have their collider disabled.

DemoPlayButton

Assets/LOADER/NewLoaderGraphics/interactable objects/DemoPlayButton.cs

Attached to each game portal prefab. When the patient gazes at / triggers it:

1
Blocks all interactables for 5 seconds: Broker<InteractablesStateChange>.Invoke(new InteractablesStateChange(5f))
2
Populates LastKnownTherapyController fields: demo_max_time = demoTime (default 30s), game_identity.game_id = (int)gameScene, game_identity.instance_id = instanceID, mode = TherapyMode.home, demo_version = true.
3
Fires TherapyEventBroker.InformSubscribers(TherapyEventFlags.Demo, mode). LoaderController responds to this flag in its ITherapyEventReceiver implementation by loading the game scene.
Note on instance ID vs game ID: instanceID is the session-set game's database row ID (unique per patient/therapy), not the game_id (which identifies the game type). The DemoTracker keys on instanceID, so demo counts are per-therapy-plan, not per-game-type globally.

StartTherapyButton

Assets/LOADER/NewLoaderGraphics/interactable objects/StartTherapyButton.cs

A 3D world-space UI button that only appears in home mode. Gate logic at Start():

  • If therapy mode is not home → SetActive(false), done
  • If any game in the session has zero demo plays → button.interactable = false, animator disabled
  • If all games have at least one demo play → button is interactive, animated

On click: calls LoaderController.Instance.StartStandaloneHomeTherapy() which sets the therapy in motion and begins the first game's 15-second countdown.

SettingsDoorInteractable

Assets/LOADER/NewLoaderGraphics/interactable objects/SettingsDoorInteractable.cs

A 3D door model the patient can hold-activate to return to the Loader (settings/home screen). The interaction is a hold-to-confirm pattern — the door rotates progressively over acceptingExitDelay seconds while the action button is held.

  • In stationary mode: BlockInteractablePerma() is called at Start() — door is locked
  • In home mode with unplayed demos: ForceDemo() calls Messenger.Broadcast("unblockDoor") to re-enable the collider so the patient can exit if needed
  • On successful activation: calls LoaderController.Instance.StartLoader()

Game Launch Flow

GameIdentity

Every game launch is identified by a GameIdentity object carrying two IDs:

class GameIdentity (from TherapyController)
intgame_idMatches ProjectScene enum integer value (7–22). Identifies which game type to load.
intinstance_idSession-set game's unique database ID. Used for REST API parameter queries (POST /therapy/getgame).

StartGame()

public async UniTaskVoid StartGame(GameIdentity gameID, LoadedSceneCallback loadedScene)
{
    if (Enum.IsDefined(typeof(ProjectScene), gameID.game_id))
    {
        currentGameID = gameID;
        ProjectScene currenGame = (ProjectScene) gameID.game_id;
        ToastText3D.Instance?.Show3DTextToast("Starting Game...");
        await AsyncLoadScene(currenGame.ToString(), loadedScene);
    }
    else
        ToastText3D.Instance.Show3DTextToast("Therapy Game Not Defined");
}

Enum.IsDefined validates the game_id against the ProjectScene enum before attempting a scene load. Unknown IDs show a toast and abort gracefully.

Parameter Polling — TherapySetParameters()

In stationary mode, after a game loads, LoaderController starts a polling loop that continuously fetches therapy parameters from the server and broadcasts them to the game:

Loop every ~1.5s while game is running
POST /therapy/getgame
fetch game params for current instance_id
Messenger.Broadcast("paramsUpdated", params)
UGCParametersController.OnParametersUpdated()
VFX applied
blur · brightness · IPD · eye patching · scaling…

The loop runs as a UniTask with a CancellationToken that fires when the game ends or the player leaves the scene. The call body uses instance_id to identify which game parameters to fetch. Exceptions are logged but the loop continues; only an explicit cancellation or network disconnect terminates it.

Home mode does not poll: In home therapy, parameters are baked into the therapy plan and do not change mid-session. TherapySetParameters is called once with repeat = false to do a single parameter fetch, or skipped if waitForParamsToStartTheGame = false.

Inter-Game Flow (Home Mode)

After a game's ModuleFinished event fires in home mode, the Hub is reloaded and HomeTherapyDelayedNextGame() is called to pace the next game start: 

public async UniTaskVoid HomeTherapyDelayedNextGame()
{
    nextGameTimerRunning = true;
    timeBetwenScenesLeft = 15f;
    Broker<InteractablesStateChange>.Invoke(this, new InteractablesStateChange(20f)); // block for 20s
    await UniTask.Delay(15000);                                                           // wait 15s
    nextGameTimerRunning = false;
    TherapyController therapyController = new TherapyController();
    therapyController.start_game = true;
    therapyController.mode = TherapyMode.home;
    therapyController.game_identity.instance_id = currentTherapy.sessionset.First()
        .sessionsetgame[homeTherapyGameIndex].id;
    therapyController.game_identity.game_id = currentTherapy.sessionset.First()
        .sessionsetgame[homeTherapyGameIndex].game.id;
    TherapyEvent(therapyController);   // triggers StartGame for next index
}

Timing

Interactables are blocked for 20 seconds via InteractablesStateChange(20f). The actual next-game start is at 15 seconds, so the block covers the full transition plus 5s buffer.

Game Index

homeTherapyGameIndex increments after each completed game. The order follows sessionset.First().sessionsetgame — set server-side by the therapist's treatment plan.

Session Completion

Assets/LOADER/Scripts/LoaderController.cs — SessionComplete(float backToLoaderDelay)

Called when all games in the therapy session have been completed. Handles data upload with retry logic, then returns the patient to the Loader.

1
Toast + half-delay: Shows "Session Completed" toast, waits backToLoaderDelay / 2 seconds (realtime).
2
Return to Hub (locked): If not already in Hub, loads Hub with InteractablesStateChange.Permanently — portals and door are frozen. This creates a "safe environment" while data uploads.
3
Save therapy data (3 retries): currentTherapy.SaveTherapy() called up to 3 times. On each retry, shows a progress toast "Sending therapy data… Attempt N/3".
4
End therapy (3 retries): Only attempted if save succeeded. currentTherapy.EndTherapy() up to 3 times with retry toasts.
5
Failure fallback: If both save + end fail, home therapy data is saved locally (SaveHomeTherapyToFile()) and a network-error dialog is shown. Unsent data can be re-submitted on next login.
6
Return to Loader: AndroidHelper.KeepScreenOff() then AsyncLoadScene(ProjectScene.Loader). Screen sleep is re-enabled after therapy ends.
Home mode only: On session complete for home therapy, MessageStore.StoreMessage("OngoingTherapy", "false") is persisted so that if the app crashes before returning to Loader, the next launch knows the therapy was fully completed and skips any "resume session" flow.

TherapyEvent Flags — Hub Relevance

TherapyController boolean fields are mapped to TherapyEventFlags bitmask values in LoaderController.TherapyEvent(). Hub-relevant mappings:

TherapyController fieldTherapyEventFlags bitHub effect
enter = true Enter Loads Hub scene, initialises therapy session
exit = true Exit Ends therapy, returns to Loader without session complete flow
start_game = true StartGame Triggers StartGame(gameIdentity) — loads the target game scene
stop_game = true StopGame Stops current game, returns to Hub
demo_version = true Demo Loads game as demo; DemoTracker increments count on ModuleFinished
force_stop = true ForceStop Immediately exits therapy; fires before scene unload
force_stop_game = true ForceStopGame Force-exits current game and returns to Hub
pause_game = true Pause Pauses game timer via UGCGameController subscription
resume_game = true ResumeGame Resumes paused game
param_changed = true ParamChanged Triggers immediate parameter re-fetch from server
continue_therapy = true ContinueTherapy Advances to next game in sequence (home mode)
(set by DemoTracker) ModuleFinished Game scene complete; triggers home-mode next-game countdown or stationary Hub return
TherapyEvent() is marked [Obsolete]. New code should dispatch events directly via TherapyEventBroker.InformSubscribers(sender, new TherapyEventArgs(flags, mode)). The old method still works and calls OldTherapyEvent(therapy) after broker dispatch.
