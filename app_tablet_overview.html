app_tablet

The clinician-facing Odroid tablet application — command & control centre for VR-based eye therapy sessions. Manages device pairing, real-time stream viewing, therapy lifecycle, and a WebView-based clinical portal.

01 Overview & Purpose

app_tablet is the clinician-side Android application installed on an Odroid tablet. It serves as the primary control panel through which a clinician monitors and manages VR-based eye therapy sessions delivered via a paired VR headset (app_gogle).

What it does
  • Displays a live mirror of the VR headset screen over TCP
  • Sends therapy commands (start, stop, pause, resume) to the headset
  • Hosts the clinical WebView portal (therapy prescriptions, patients, reports)
  • Manages clinician login, patient records, and therapy sessions in a local SQLite DB
  • Discovers and pairs with VR headsets on the local network
  • Uploads recorded therapy sessions to the remote backend
Module dependencies
  • base — shared networking, streaming, NSD, PermissionUtils, BaseService
  • ORMLite 6.1 — SQLite ORM (entities, DAOs)
  • Volley — HTTP request queue
  • tus-java-client — Resumable upload protocol
  • Firebase Crashlytics + AppCenter — Crash analytics
  • Gson + Joda-Time — Serialization / date handling

02 Architecture

app_tablet follows an Application-as-Hub pattern with no dependency injection framework. The RemmedApplication singleton holds references to active view interfaces and routes data from background services to the UI. Interface-driven contracts decouple UI from business logic.

03 Application Class — RemmedApplication

RemmedApplication is the singleton coordinator of the entire app. It initialises all subsystems on startup and acts as the message bus between background services and the active UI.

Startup Responsibilities
  • Initialise DatabaseHelper (ORMLite)
  • Set SERVER_ADDRESS and WEB_SOCKET_ADDRESS from build variant
  • Configure Firebase Crashlytics & AppCenter analytics
  • Load current user via LocalUserManager
  • Register CrashAppHandler as global UncaughtExceptionHandler
  • Reset Gogle IP on first run
Routing Methods
  • setStreamFrame(Bitmap) — routes video frame to active IStreamView
  • setEyetrackerStreamFrame(Bitmap) — routes eye-tracker frame
  • connectionInfo(boolean) — notifies UI of Gogle connect/disconnect
  • receivedGogleResponse(GogleReceiveData) — routes parsed JSON response to IGogleResponse

05 Services

ℹ
All services extend BaseService from the base module. They use START_STICKY so the OS restarts them if killed, and run as foreground services with persistent notifications. Static volatile fields (e.g. IS_RUNNING, IS_STREAMING) expose runtime state to the rest of the app.
Service Port / Connection Direction Purpose
DownstreamService 6776 TCP ← Headset Receives JPEG-compressed video frames; decompresses to Bitmap; queues up to 30 frames; calculates FPS; feeds RemmedApplication.setStreamFrame()
DownstreamEyetrackerService 6778 TCP ← Headset Same as DownstreamService but for the eye-tracker data stream
GogleSendDataService 8502 TCP → Headset Maintains a persistent socket; sends JSON therapy commands (start/stop/pause); includes ping keepalive mechanism
GogleReceiveDataService 8503 TCP ← Headset Listens for JSON status/telemetry responses; deserialises GogleReceiveData; routes via RemmedApplication
WebServerService 8080 HTTP (local) ↔ WebView Runs a local HTTP proxy; WebView loads localhost:8080; server forwards requests to the configured backend, enabling mixed-content and cookie injection
UpstreamWebService HTTPS backend → Server Sends recorded stream data to the remote backend server
UploadStreamWebService HTTPS (tus) → Server Resumable therapy recording upload using tus-java-client; survives network interruptions
ZipService Local storage internal Compresses therapy recording files in background; notifies UI on completion

07 Video Streaming

The VR headset captures its display, compresses each frame as JPEG, and sends a continuous stream over TCP. The tablet receives, decompresses, and renders each frame. 


  ┌─────────────────────────────────────────┐
  │  app_gogle — UpstreamService            │
  │  Capture display → compress JPEG bytes  │
  └─────────────────┬───────────────────────┘
                    │ TCP port 6776
                    ▼
  ┌──────────────────────────────────────────────────────────────────┐
  │  DownstreamService                                                │
  │  1. ServerSocket.accept() on port 6776                           │
  │  2. while (running) { read DataInputStream → byte[] frame }      │
  │  3. BitmapFactory.decodeByteArray() → Bitmap                     │
  │  4. bytesReceivedStream.add(bitmap)   // max 30 frames queued    │
  │  5. StreamUtils.calculateFPS(frames, duration_ms)                │
  └─────────────────────────────┬────────────────────────────────────┘
                                │ Bitmap
                                ▼
  ┌───────────────────────────────────────────────────────┐
  │  RemmedApplication.setStreamFrame(Bitmap)             │
  │  → active IStreamView.setStreamFrame(bitmap)          │
  └───────────────────────────────┬───────────────────────┘
                                  │ runOnUiThread
                    ┌─────────────┴───────────────┐
                    ▼                             ▼
  ┌─────────────────────────┐    ┌───────────────────────────────┐
  │  HomeActivity            │    │  FullscreenStreamActivity     │
  │  ImageView (preview)     │    │  ImageView (full resolution)  │
  └─────────────────────────┘    └───────────────────────────────┘

FPS Calculation

StreamUtils computes FPS as: 

fps = (frames / duration_ms) * 1000

The eye-tracker stream follows an identical path via DownstreamEyetrackerService on port 6778.

A bidirectional JSON command channel operates alongside the video stream. The tablet sends commands on port 8502 (Gogle's receive port) and listens for responses on port 8503 (Gogle's send port).

— Local Web Server (WebServerService)

The WebView loads http://localhost:8080. WebServerService runs a lightweight HTTP server that proxies all requests to the configured backend server. This approach solves mixed-content restrictions and allows the server to inject session cookies transparently. 


  RemmedWebView → http://localhost:8080/some/path
                        │
                        ▼
              WebServerService (port 8080)
                        │ forward request + inject headers/cookies
                        ▼
              Backend Server (e.g. dev.remmedvr.com)
                        │ response
                        ▼
              WebServerService → response → WebView

WebServerUtils handles URL parameter parsing. WebViewParamsUtils manages WebView-specific parameter extraction.

09 WebView & JavaScript Bridge

RemmedWebView is a custom AdvancedWebView subclass that hosts the clinical portal. A JavascriptInterface class (52 methods) provides bidirectional communication between the HTML/JS frontend and Android Java.

RemmedWebView Configuration

  • JavaScript enabled, DOM storage enabled
  • Universal file access enabled (cross-origin)
  • Mixed content allowed (HTTP within HTTPS)
  • No user gesture requirement for media playback
  • SSL errors suppressed in non-production builds
  • reloadWebviewRunnable — auto-retry on load failure

⚠

11 Manager Classes

Managers encapsulate domain-specific business logic and state. Local managers operate on device state; Remote managers wrap HTTP calls.

Local Managers
  • LocalTherapyManager — therapy state, session limits, local caching
  • LocalUserManager — current user state, login persistence, role checks
  • LocalPairingManager — Gogle pairing state, stored credentials
  • LocalCacheManager — generic local key-value cache operations
  • GogleAvailabilityManager — Gogle reachability and status tracking
  • ConnectingGogleProgressManager — UI progress state during connection
  • AppWallpaperManager — dynamic wallpaper per therapy type
  • CellLocationManager — cellular location tracking
Remote Managers (HTTP)
  • RemoteTherapyManager — sync therapies from backend

  • RemoteUserManager — user auth and profile sync
  • RemoteWebLogManager — send client-side logs to server
  • RemoteDeviceTypeManager — fetch device type catalog
