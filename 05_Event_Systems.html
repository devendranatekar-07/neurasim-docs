Event Systems

Three event systems coexist in the codebase. Each serves a different scope and purpose. New code should use HierarchicalBroker for system-wide events and TherapyEventBroker for therapy state commands.

Overview

Obsolete  Messenger

String-keyed global event bus. Predates the type-safe systems. Still widely used in ComboManagers, parameter broadcasting, and game events — but new code should not add more usages.

Preferred  HierarchicalBroker

Generic type-safe event system. No engine dependencies. Supports pre-event hooks with cancellation, post-event hooks, and structured logging. Use for all new system-wide events.

Therapy  TherapyEventBroker

Specialised static broker for therapy state commands (Enter, StartGame, Pause…). Priority-sorted subscribers. Dispatch stops early if a subscriber returns false.

Messenger [Obsolete]

Assets/Shared/Scripts/Helpers/Messenger/Messenger.cs
Do not add new Messenger usages. The Messenger class is marked [Obsolete]. It exists to support legacy game code (ComboManagers, parameter broadcasting via "paramsUpdated"). New events should use Broker<T> instead.

Messenger is a static singleton backed by an eventTable dictionary mapping string keys to Delegate lists. Supports callbacks with 0, 1, 2, or 3 generic parameters.

API

AddListener / RemoveListener

// No params
Messenger.AddListener("key", callback);

// One param
Messenger<int>.AddListener("key", (val) => { });

// Two params
Messenger<int, string>.AddListener("key", (a, b) => { });

// Remove
Messenger.RemoveListener("key", callback);

Broadcast

// No params
Messenger.Broadcast("key");

// One param
Messenger.Broadcast("key", value);

// Two params
Messenger.Broadcast("key", a, b);

// MessengerMode controls error policy:
// REQUIRE_LISTENER | DONT_REQUIRE_LISTENER

Permanent Messages

Messenger keys can be marked as permanent, meaning they survive the scene-load cleanup that normally purges all listeners. This is used for cross-scene subscribers like UGCParametersController, which must keep receiving "paramsUpdated" even after a scene transition. 

// Mark a key permanent so it survives scene changes:
Messenger.MarkAsPermanent(LoaderController.ParamsUpdated);

// Add a permanent listener (in UGCParametersController.Awake):
Messenger<List<Param>>.AddListener(
    LoaderController.ParamsUpdated,
    OnParametersUpdated,
    MessengerMode.DONT_REQUIRE_LISTENER
);

Cleanup on Scene Load

MessengerHelper is a hidden MonoBehaviour (created automatically) that subscribes to SceneManager.sceneLoaded. On each scene load it calls Messenger.Cleanup(), which removes all non-permanent listeners from the event table. This prevents stale listeners from firing in the wrong scene.

Memory leak risk: If you add a Messenger listener and forget RemoveListener, the listener will be cleaned up on scene load — but only if the key is not marked permanent. For permanent keys, you must remove the listener manually to avoid leaks.

HierarchicalBroker Preferred

Assets/Shared/Scripts/HierachicalBroker/

A generic, engine-independent event system. Events are typed via an IEventArgs marker interface. Each event type has its own static root broker accessed via Broker<T>. Subscriptions return IDisposable — disposing automatically unsubscribes.

Interfaces

IEventArgs

IEventArgs.cs 
// Marker interface — all event arg types implement this
public interface IEventArgs { }

IBroker<T>

IBroker.cs 
public interface IBroker<T> where T : IEventArgs
{
    delegate void Delegate(object sender, in T args);
    IDisposable Subscribe(Delegate @delegate);
    IBeforeBroker<T> Before { get; }
    IBroker<T> After { get; }
}

IBeforeBroker<T>

IBeforeBroker.cs 
public interface IBeforeBroker<T> where T : IEventArgs
{
    delegate void Delegate(
        object sender,
        in T args,
        ref bool cancel);  // set true to cancel

    IDisposable Subscribe(Delegate @delegate);
    IBeforeBroker<T> Before { get; }
    IBroker<T> After { get; }
}

Subscribe & Invoke

Subscribe

// Returns IDisposable — dispose to unsubscribe
IDisposable sub = Broker<TabletStateChanged>
    .Subscribe(OnTabletStateChanged);

void OnTabletStateChanged(
    object sender,
    in TabletStateChanged args)
{
    // handle event
}

// Unsubscribe (e.g., in OnDestroy):
sub.Dispose();

Invoke

// Fire event from any sender
Broker<TabletStateChanged>.Invoke(
    this,
    new TabletStateChanged(
        address, ClientState.Connected));

// Real example from TabletCommunication:
Broker<BatteryLevelChanged>.Invoke(
    this,
    new BatteryLevelChanged(level));

Before / After Hooks

Every broker has a .Before chain and an .After chain. These allow pre- and post-processing of events without modifying the primary subscriber.

Broker<T>.Before.Subscribe(...)
pre-event hook — can cancel
cancel == true?
Yes — cancelled
Invoke aborted
main handler NOT called
No — proceed
Broker<T>.Subscribe(...)
main handler fires
Broker<T>.After.Subscribe(...)
post-event hook
// Pre-event hook — can cancel the invoke
Broker<InteractablesStateChange>.Before
    .Subscribe((object sender, in InteractablesStateChange args, ref bool cancel) =>
    {
        if (isLocked) cancel = true;  // abort the event
    });

// Post-event hook — fires after main handlers
Broker<ObjectSizeChanged>.After
    .Subscribe(OnObjectSizeChangedAfter);

Cancellation

Only .Before subscribers can cancel an event. Setting cancel = true prevents the main IBroker<T> subscribers and the .After chain from firing. The BeforeBroker logs a cancel event if a [BrokerLogger] attribute is present on the args type.

Logging

Attach [BrokerLogger(typeof(MyLogger), LoggedEvents.All)] to an event args class to enable structured logging of Subscribe, Invoke, Unsubscribe, and CancelInvoke calls. The logger must implement IBrokerLogger<T>. 

// Attribute on the event args class:
[BrokerLogger(typeof(TabletStateChangedLogger), LoggedEvents.Invoke | LoggedEvents.Subscribe)]
public class TabletStateChanged : IEventArgs { ... }

// LoggedEvents flags enum:
[Flags]
public enum LoggedEvents
{
    None = 0, Subscribe = 1, Invoke = 2,
    Unsubscribe = 3, CancelInvoke = 4,
    All = Subscribe | Invoke | Unsubscribe | CancelInvoke
}

TherapyEventBroker Therapy Commands

Assets/Shared/Scripts/TherapyEvents/

A static class that dispatches therapy state commands to a priority-sorted list of ITherapyEventReceiver subscribers. Unlike Broker<T>, dispatch stops early if any subscriber returns false — allowing watchdogs to intercept and block therapy state transitions before they reach LoaderController.

TherapyEventFlags

Assets/Shared/Scripts/TherapyEvents/TherapyEventFlags.cs

A [Flags] bitmask enum. Multiple flags can be set in a single event. LoaderController.TherapyEvent() maps TherapyController boolean fields to these flags before dispatching.

ITherapyEventReceiver

Assets/Shared/Scripts/TherapyEvents/ITherapyEventReceiver.cs 
public interface ITherapyEventReceiver
{
    // Bitmask of flags this receiver wants to handle.
    // Broker only calls OnTherapyEvent if (incoming & AcceptedEvents) != 0.
    public TherapyEventFlags AcceptedEvents { get; }

    // Return true  → allow dispatch to continue to lower-priority subscribers.
    // Return false → halt chain — no further subscribers are called.
    public bool OnTherapyEvent(object sender, TherapyEventArgs args);
}

// Example: UGCGameController only handles Pause and ResumeGame
public TherapyEventFlags AcceptedEvents =>
    TherapyEventFlags.Pause | TherapyEventFlags.ResumeGame;

Priority System

Assets/Shared/Scripts/TherapyEvents/TherapyEventBrokerKnownSubscribers.cs

Subscribers are stored in a SortedList<int, ITherapyEventReceiver> keyed by priority integer. Lower integer = higher priority = called first. Watchdogs sit at the top of the chain and can block events before they reach LoaderController.

Reserved range: Priorities from -2048 to 1024 are reserved for known subscribers. Anonymous subscribers (no explicit priority) are auto-assigned an incrementing value outside this range.
SubscriberPriorityCalledPurpose
BatteryLevelWatchdog−2051 (highest)FirstBlock therapy entry if headset battery too low
NetworkStatusWatchdog−2050SecondBlock therapy entry if no network connectivity
InactivityWatchdog−2049ThirdBlock therapy entry on device inactivity

// Subscribe with a known priority:
TherapyEventBroker.SubscribeToTherapyEvents(
    this,
    TherapyEventBrokerKnownSubscribers.LoaderController);

// Subscribe without a priority (auto-assigned):
TherapyEventBroker.SubscribeToTherapyEvents(this);

// Unsubscribe:
TherapyEventBroker.UnsubscribeFromTherapyEvents(this);

Choosing a System

Use Messenger when…

  • Modifying existing game code that already uses it
  • Broadcasting "paramsUpdated" (no alternative yet)
  • Firing "correctAnswer" / "wrongAnswer" in a ComboManager

Do not add new Messenger keys for new features.

Use Broker<T> when…

  • Building any new system-wide or cross-scene event
  • You need compile-time type safety
  • You need pre-event validation or cancellation
  • You want structured logging
  • The event is not a therapy state command

Use TherapyEventBroker when…

  • Responding to therapy state transitions (Enter, StartGame, Pause…)
  • Implementing a watchdog that can block therapy entry
  • You need priority-ordered dispatch
