Therapeutic VFX & Difficulty

How therapy parameters drive per-eye visual effects, and how player performance drives difficulty and object scaling.

1. How the Systems Connect

These two systems sit at the output end of the therapy pipeline. Parameters arrive from the dashboard, the VFX system translates them into on-screen changes (blur, brightness, eye patching, IPD), and the difficulty system adapts those parameters in real time based on how well the patient is performing.

Dashboard / Tablet sends therapy parameters

Visual params (IPD, blur, etc.)

objectSize / difficulty

Key relationship: UGCTherapeuticEventsController handles the therapist-prescribed visual effects (fixed per session). ComboManager drives the adaptive difficulty (changes live during gameplay). Both write through UGCParametersController.

2. UGCTherapeuticEventsController

Assets/UGC-Protype/UGCTherapeuticEventsController.cs

Singleton MonoBehaviour present in every game scene. It subscribes to parameter callbacks on Start() and translates each incoming value into a rendering change on the left or right eye camera. It is the only place in the codebase that touches camera-level image effects.

2.1 Eye Patching

Eye patching controls which eye can see each group of game objects. It works by changing the Unity layer of every EyePatchable in the scene to match the dominant-eye parameter.

Assets/UGC-Protype/EyePatchable.cs

Place EyePatchable on any GameObject that needs per-eye visibility control. Set its group to one of:

Interactable

The objects the patient interacts with — targets, collectibles, response stimuli.

Deployer

Objects that spawn or host game elements — launchers, containers, background actors.

Controller

The aiming reticle / controller pointer visible to the patient.

Each group has its own transparencyParam and eye-layer assignment in UGCTherapeuticEventsController, so the therapist can independently patch each group to a different eye.

EyeLayer enum — Assets/Shared/Scripts/Shared Scripts/SharedHelper.cs
int 10LeftEyeObject rendered only by the left eye camera (Camera culling mask layer 10)
int 11RightEyeObject rendered only by the right eye camera (Camera culling mask layer 11)
int 0BothDefault layer — visible to both eyes

EyePatchable.Patch(EyeLayer) recursively walks child transforms up to depth levels (default 10) and sets each GameObject.layer. It re-patches automatically on OnEnable().

2.2 IPD Management

Interpupillary distance is applied as a horizontal camera offset. The value arrives as an integer (millimetres) from the currentIPDParam and is converted to world-space units:

void SetIPD(int ipd_mm)
{
    float offset = (ipd_mm / 2f) / 1000f;          // mm → metres, half per eye
    leftCamera.localPosition  = new Vector3(-offset, 0, 0);
    rightCamera.localPosition = new Vector3(+offset, 0, 0);
}

Default IPD is 65 mm. Changing this in-session does not require a scene reload.

2.3 Visual Effects

All effects are applied per-eye via two render pipeline components: ColorGradingLwrp (URP renderer feature, per-eye gamma/contrast) and StandardRP_Blur_Brightness (singleton managing BlurOptimized and BrightnessContrastGamma image effects on each camera).

Which eye receives which effect? Effects are intentionally asymmetric — therapeutic amblyopia/suppression treatment applies stimulation to the weaker eye and patching/reduction to the dominant eye. The dominant eye is determined by dominantEyeParam.

2.4 Object Scaling (Difficulty-Driven)

When difficulty changes, OnDifficultyChange(int difficulty) recalculates a scale value from the current ObjectSize mode and broadcasts it to all Scalable components in the scene.

// Scale formula: baseSize - (difficulty × 0.03)
float scale = objectSizeParam.FirstValue switch
{
    ObjectSize.Small  => 0.7f - difficulty * 0.03f,
    ObjectSize.Medium => 1.0f - difficulty * 0.03f,
    ObjectSize.Big    => 1.3f - difficulty * 0.03f,
    _ => 1.0f
};

3. ComboManager & Difficulty

Assets/Combos/ComboManager.cs

ComboManager is an abstract MonoBehaviour that sits in each game scene and listens to three Messenger events. It accumulates a difficulty-progress score and, once thresholds are crossed, increments or decrements the global difficulty level (0–10).

Listens for

  • Consts.correctAnswer — patient answered correctly
  • Consts.wrongAnswer — patient answered incorrectly
  • Consts.timeout — answer window expired

Broadcasts

  • Consts.onDifficultyChange — received by UGCTherapeuticEventsController to rescale objects

3.1 Difficulty Progression Mechanics

Difficulty does not change immediately on each event. Instead a static accumulator (difficultyProgress) collects deltas, and only crosses thresholds trigger a level change:

Correct answer
+1 progress
Wrong answer
−1 or −2 progress
Timeout
varies by subclass
difficultyProgress accumulates
progress > 5
difficulty++ (max 10), reset to 1
progress < 0
difficulty-- (min 0), reset to 4
otherwise
no level change
Messenger.Broadcast("onDifficultyChange", difficulty)
UGCTherapeuticEventsController rescales all Scalable objects
Why an accumulator instead of direct change? The threshold system creates hysteresis — difficulty does not oscillate rapidly on alternating correct/wrong answers. The patient must demonstrate sustained improvement (>5 accumulated) before the game gets harder, and sustained failure before it gets easier.

The reset values after a threshold crossing (1 after going up, 4 after going down) give the system a head-start toward the next change in the same direction, rewarding streaks.

3.2 Combo Mechanic

Alongside difficulty progression, ComboManager tracks a combo level that unlocks celebratory visual feedback via a ComboActivate component. Combos are purely cosmetic — they do not affect difficulty directly.

Combo lifecycle
step 1Combo(comboLevel)Called by subclass when a threshold of correct answers is reached. Triggers combo.StartCombo(level) visual.
step 2OnReachingMaxComboIf level ≥ 2, starts a 5-second coroutine before automatically breaking the combo.
step 3BreakCombo()Called on wrong answer, timeout, or after 5-second max-combo timer. Resets combo visual and combo counter.

Subclasses define the threshold for invoking a combo — typically a list of correct-answer counts per level (e.g., [3, 5, 7] means 3 correct for level 1, 5 more for level 2, 7 more for level 3).

3.3 Scalable Component

Assets/UGC-Protype/Scalable.cs

Place Scalable on any GameObject that should respond to difficulty changes. UGCTherapeuticEventsController.OnDifficultyChange() calls FindObjectsOfType<Scalable>() and broadcasts the new scale to each one.

Scalable — key fields
boolrelativeScaleIf true, multiplies the incoming scale against the object's starting scale rather than Vector3.one
boolscaleGraduallyOverTimeAnimates the scale change over scaleTime seconds (default true)
floatscaleTimeDuration of the scale animation in seconds (default 0.5s)
boolautoScaleOnParamChangeIf false, the object opts out of automatic difficulty-driven rescaling

On OnEnable(), Scalable calls ScaleFromZero() — the object animates in from zero scale, giving a smooth appearance effect at spawn. The GetFinalScale() virtual method can be overridden in subclasses for per-object scale adjustments.

// Gradual scale coroutine rechecks param.SecondValue each frame,
// so if difficulty changes mid-animation it catches up cleanly.
while (elapsed < scaleTime)
{
    elapsed += Time.deltaTime;
    float t = elapsed / scaleTime;
    transform.localScale = Vector3.Lerp(from, baseScale * param.SecondValue, t);
    yield return null;
}

IncrementalComboManager — Inspector Configuration

The key Inspector field is n_CorrectAnswersToInvokeCombo — a list where each element is the number of additional correct answers needed to reach that combo level:

// Example: [3, 5, 7]
// Level 0→1: 3 correct answers
// Level 1→2: 5 more correct answers
// Level 2→3: 7 more correct answers → OnReachingMaxCombo() → auto-break after 5s

keepComboBetweenLevels (default false) — if false, the combo counter resets when a new level starts in the scene.

FollowGameComboManager — Inspector Configuration

Uses followTimesToInvokeCombo — a list of cumulative seconds the patient must keep the reticle on the target to advance each combo level:

// Example: [3.5, 7.2, 12.0] seconds of continuous tracking

Every second of successful tracking adds +1 difficulty progress. Breaking tracking calls SetDifficulty(-2).
