NeuraSim VR

Complete technical overview of the BeeVee VR vision therapy platform — architecture, systems, and data flow.

1. Introduction

NeuraSim VR (product name: BeeVee) is a VR-based vision therapy application. It delivers interactive therapy games on Android VR headsets (Pico Neo, DPVR), controlled either by a therapist via a companion tablet (stationary mode) or by the patient independently (home mode). Game parameters, difficulty, and visual effects are dynamically controlled from a cloud dashboard.

Detailed documentation available The login/authentication flow has its own dedicated document: 01_Login_Flow.html. Below is a summary.
Authentication Model

NeuraSim uses device-centric authentication. The VR headset's serial number is pre-registered in the backend by a clinic administrator. No username or password is needed from the patient.

POST /web/login
api: static key · rkey: TOTP (SHA-256, 60s) · uuid: device serial (Pico SDK)

Post-login credentials: The token_session.token (JWT-like) is stored in WebApiPass and used for all subsequent API calls (/therapy/get, /therapy/start, /therapy/end, /therapy/save).

4.2 Therapy Modes

Stationary Mode

Therapist-controlled clinic sessions. Tablet directs all game selection, parameter changes, pause/resume. TCP servers active on ports 8502/8503. Communication mode: Tablet.

Tablet → VR therapy commands, params, init_data

VR → Tablet telemetry, game results, session status

Home Mode

Patient self-directed sessions. No tablet needed. App fetches therapy plan from server on login, patient plays games independently. Communication mode: Standalone. TCP servers stopped.

Daily limit therapy.sessionsperday enforced

Unsent data resent on next login via CheckAndResendPreviousTherapy()

4.3 Real-Time Parameter Polling

Assets/LOADER/Scripts/LoaderController.cs — PollTherapyParams()

During active gameplay, LoaderController polls the server for parameter updates every ~1.5 seconds. When new parameters arrive, they are broadcast to all game components:

// Polling loop (simplified)
while (gameIsRunning)
{
    var response = await RestApiCall.PostRequest("/therapy/getgame", callBody);
    Messenger.Broadcast("paramsUpdated", response.game.parameters);
    await UniTask.Delay(1500);
}

5. Game Framework (UGC)

Assets/UGC-Protype/

The UGC namespace provides the base framework every therapy game is built upon. Each game scene must contain three core components working together:

UGCGameController

Singleton managing game lifecycle: start, pause, resume, timer. Holds references to cameras, parameters, levels, and therapeutic events controllers.

UGCParametersController

Receives therapy parameters via Messenger. Maintains shared static params + game-specific ScriptableParameter[] from Inspector.

UGCTherapeuticEventsController

Subscribes to all parameter OnUpdate callbacks. Applies visual effects: IPD, blur, brightness, contrast, eye patching, transparency, scaling.

Game-specific parameters are ScriptableObject assets assigned to the scene's UGCParametersController.scriptableParameter[] array in the Inspector. Types: BoolParameter, IntParameter, BackgroundModeParameter. Created via Unity menu: Parameters/Bool, Parameters/Int.

7. Therapeutic Visual Effects

Assets/UGC-Protype/UGCTherapeuticEventsController.cs

The therapeutic VFX system applies per-eye visual modifications driven by parameters from the dashboard. This is the core of the vision therapy — controlled visual stress to train the patient's eyes.

7.1 Eye Patching (Layer-Based)

Assets/UGC-Protype/EyePatchable.cs

Objects with EyePatchable component can be shown to one eye or both, using Unity's layer system. The left and right eye cameras render different layer masks.

EyeLayerEffect
BothObject visible to both eyes (normal)
LeftEyeObject only rendered by left eye camera
RightEyeObject only rendered by right eye camera

Three separate patching channels control different object groups independently:

  • aimingCamParam — Crosshair / pointer reticle
  • interactableObjectCameraParam — Game objects the patient interacts with
  • deployerObjectCameraParam — Spawners / sources of game objects

7.2 IPD (Interpupillary Distance)

Software-controlled stereo camera separation. Default 65mm, adjustable by therapist. Applied by offsetting left/right camera X positions: leftCam.localPosition.x = -(ipd/2)/1000, rightCam.localPosition.x = +(ipd/2)/1000.

7.3 Blur, Brightness, Darkness, Contrast

Post-processing effects applied independently per-eye to create controlled visual imbalance:

EffectApplied ToRangeTherapy Goal
BlurNon-dominant eye0-100Force dominant eye reliance → amblyopia training
BrightnessNon-dominant eye0-100Increase visibility of weaker eye
DarknessDominant eye-100 to 0Reduce dominant eye advantage
ContrastPer-eye configurable0-100Fine-tune visual balance
TransparencyObjects0-100 → 0.1-0.8Reduce object salience for specific eye

9. Event Systems

Three event systems coexist in the codebase. New code should prefer Broker<T> for system-wide events and TherapyEventBroker for therapy commands.

9.1 Messenger (Legacy — [Obsolete])

Assets/Shared/Scripts/Helpers/Messenger/Messenger.cs

String-keyed global event bus. Still widely used in ComboManagers, parameter broadcasting, and game events.

KeyPayloadUsed By

"wrongAnswer"objectGame → ComboManager

"levelCompleted"objectLevels → Results

9.2 HierarchicalBroker (Preferred)

Assets/Shared/Scripts/HierachicalBroker/

Type-safe generic event system. Engine-independent (noEngineReferences: true). Supports Before/After hooks and cancellation.

// Subscribe (returns IDisposable for auto-cleanup)
var sub = Broker<ObjectSizeChanged>.Subscribe((sender, args) => {
    Debug.Log($"Size: {args.Size}");
});

// Invoke
Broker<ObjectSizeChanged>.Invoke(this, new ObjectSizeChanged(size));

// Before hook (can cancel event)
Broker<ObjectSizeChanged>.Before.Subscribe((sender, args) => { /* validate */ });

Event types used: TabletStateChanged, ObjectSizeChanged, ObjectDistanceChanged, NewTranslation, UpdatingStateChanged, BatteryLevelChanged, ControllerStateChanged, InteractablesStateChange.

9.3 TherapyEventBroker

Assets/Shared/Scripts/TherapyEvents/

Priority-sorted event dispatcher for therapy commands. Subscribers implement ITherapyEventReceiver and can halt event propagation by returning false.

TherapyEventFlag (bitmask)Trigger
EnterTherapy session starts
ExitTherapy session ends
StartGameLoad a game scene
StopGameStop current game
ForceStopEmergency stop entire therapy
PausePause active game
ResumeGameResume paused game
ParamChangedParameters updated mid-game
DemoDemo/preview mode activated
ModuleFinishedGame module completed

Key subscribers: LoaderController (priority orchestrator), UGCGameController (pause/resume), NetworkStatusWatchdog (blocks home therapy if offline), BatteryLevelWatchdog (blocks Hub entry on low battery).

10.2 Device Discovery

The VR headset broadcasts its presence for the tablet to find it on the network:

  • UDP broadcast on port 8505 every 3 seconds: "NEURASIM_VR|{IP}|{PORT}|{deviceName}|{deviceId}"
  • Multicast to 224.0.0.1 every 5 seconds (Android 8.1+ compatibility)

14. VR-Specific Systems

Stereo Rendering

URP with separate left/right eye cameras. Layer-based eye patching controls which objects each eye sees. VFX applied per-eye via post-processing.

Angular Sizing

AngularSizeable system maintains consistent visual angles regardless of depth. VisusController manages visual acuity levels.

Controller Abstraction

UGCInput, RemmedControllers abstract across Pico 3DoF/6DoF and DPVR. States: Enabled, EnabledWithPointers, Disabled. State changes on pause, UI mode, and during OTA update installs.

Thread Marshalling

UnityThread.executeInUpdate() dispatches TCP background thread callbacks to Unity main thread. Critical for tablet communication handlers.

BatteryLevelWatchdog

Persistent singleton. Monitors headset + controller battery levels continuously. Fires Broker<BatteryLevelChanged>. Blocks Hub entry if battery below minimum. Shows blocking dialog to the patient if levels are too low.

AndroidHelper

Static utility class wrapping Android-specific operations: WiFi connection state, silent APK installs via remmedService, device wake-up, screen brightness control, and environment setup on first launch.
